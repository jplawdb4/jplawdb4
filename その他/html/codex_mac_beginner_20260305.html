Section 01 — Overview

Codex Mac版とは何か — 概要・位置づけ・CLI版/クラウド版との違い

Codex Mac版アプリの概要

OpenAI Codex Mac版デスクトップアプリ（以下「Codex App」）は、2026年2月2日にリリースされた macOS ネイティブのデスクトップアプリケーションです。OpenAI は公式にこれを「エージェントのためのコマンドセンター」と位置づけており、複数の AI コーディングエージェントを並列に管理・実行し、長時間にわたるタスクを協調的に処理するための専用インターフェースとして設計されています。

Codex は2025年4月にクラウドベースのコーディングエージェントとして初めてリリースされ、その後 CLI（コマンドラインインターフェース）版、IDE 拡張機能版が続きました。そして2026年2月、これらを統合的に管理するための「デスクトップアプリ」として Mac 版が登場しました。単なるラッパーではなく、マルチタスク管理、Worktree によるブランチ分離、Automations（自動化）、Skills（スキル）管理、統合ターミナル、差分レビューなど、デスクトップアプリならではの機能を豊富に備えています。

2026年3月4日時点では Windows 版もリリースされていますが、本ガイドでは Mac 版に焦点を当てて解説します。

CLI版・クラウド版・Mac版の位置づけの違い

OpenAI の Codex エコシステムには現在、大きく3つの利用形態があります。それぞれの役割と特徴を理解しておくことが重要です。

Codex CLI（ターミナル版）

ターミナル上で動作するコマンドラインツールです。codex コマンドを使い、カレントディレクトリのコードに対してエージェントが読み取り・編集・コマンド実行を行います。オープンソースで公開されており、npm 経由でインストールできます。ローカルマシンのサンドボックス内で動作し、開発者がターミナルから離れずに作業できるのが特徴です。1つのセッションで1つのタスクを実行するシングルスレッド型のワークフローが基本となります。

Codex Cloud（ChatGPT内クラウド版）

ChatGPT のウェブインターフェースや iOS アプリから利用できるクラウドベースの Codex です。GitHub リポジトリと連携し、OpenAI が管理する隔離コンテナ内でエージェントが動作します。ローカル環境を汚さずにタスクを実行でき、自動コードレビューや Slack 連携などのクラウド専用機能が利用可能です。ただし、ローカルファイルに直接アクセスすることはできません。

Codex App（Mac版デスクトップアプリ）

今回解説するデスクトップアプリです。CLI の機能をベースにしつつ、GUI による直感的な操作、複数エージェントの並列管理、プロジェクトごとのスレッド整理、Git Worktree のネイティブサポート、差分レビュー機能、Automations（定期自動実行）、Skills の管理 UI、統合ターミナル、IDE 拡張との同期など、開発者の日常ワークフローを大幅に強化する機能を提供します。CLI と設定ファイルやセッション履歴を共有しているため、既存の CLI ユーザーはシームレスに移行できます。

ポイント: Mac版アプリは CLI の上位互換ではなく、補完的なツールです。CLI のパワーユーザーは引き続きターミナルを使いつつ、複数エージェントの管理やレビュー・自動化のためにアプリを併用する、というワークフローが推奨されています。

なぜデスクトップアプリが必要だったのか

OpenAI の公式ブログでは、既存の IDE やターミナルベースのツールは「複数のエージェントを並列で指示・監督・協調させる」という新しい開発スタイルをサポートするために設計されていない、という課題が述べられています。Codex リリース以降、開発者の働き方は根本的に変化し、モデルが複雑で長時間にわたるタスクを端から端まで処理できるようになった結果、「エージェントに何ができるか」ではなく「どうやって複数のエージェントをスケールして管理するか」が中心的な課題になりました。

Codex App はこの課題に対する回答として設計されており、プロジェクトごとにスレッドを整理し、エージェントの変更を差分で確認し、コメントを付けてフィードバックし、必要に応じてエディタで手動修正を行い、さらにバックグラウンドで自動化タスクを走らせる —— こうした統合的なエージェント管理体験を提供しています。

Section 03 — Installation

インストール方法（ダウンロード・セットアップ手順）

GUI によるインストール（推奨）

  1. 公式サイトから DMG をダウンロード

    ブラウザで developers.openai.com/codex/app/ にアクセスし、「Download for macOS」ボタンをクリックします。約 25 MB の DMG ファイルがダウンロードされます。

  2. DMG を開いてインストール

    ダウンロードした Codex.dmg をダブルクリックして開き、表示される Codex アイコンを /Applications フォルダにドラッグ&ドロップします。標準的な Mac アプリのインストール手順と同じです。

  3. アプリを起動してサインイン

    Launchpad や Spotlight（⌘ + Space → 「Codex」と入力）からアプリを起動します。初回起動時にサインイン画面が表示されるので、ChatGPT アカウントまたは OpenAI API キーでログインします。

  4. プロジェクトを選択

    サインイン後、作業対象のプロジェクトフォルダを選択します。過去に CLI や IDE 拡張で使用したプロジェクトがある場合、自動的に表示されます。

  5. 最初のメッセージを送信

    プロジェクトを選択し「Local」モードが選ばれていることを確認したら、テキスト入力欄にタスクや質問を入力して送信します。これで Codex がローカルマシン上でエージェントとして動作を開始します。

CLI 経由でのインストール

既に Codex CLI がインストールされている場合、ターミナルから以下のコマンドでデスクトップアプリを起動することもできます。

codex app

このコマンドは、Codex App がインストールされていればそれを起動し、未インストールの場合はダウンロードページへの案内を表示します。CLI と App は設定ファイル（~/.codex/config.toml）やセッション履歴を共有しているため、CLI で設定した内容がアプリにも反映されます。

サインイン方法の選択

Codex App では2種類のサインイン方法が用意されています。

方法 特徴 利用可能な機能
ChatGPT アカウント 通常の ChatGPT ログイン（Google / Apple / メール等）を使用 すべての機能（ローカル、クラウド、コードレビュー、Slack 連携等）
OpenAI API キー API キーを入力してログイン ローカルタスクのみ。Cloud スレッドやクラウド連携機能は利用不可。API 従量課金。

一般的な開発者には ChatGPT アカウントでのサインインが推奨されます。API キーでのサインインは、CI/CD パイプラインでの自動化や、組織の共有環境での利用に適しています。

Section 04 — Features

主な機能と特徴

マルチプロジェクト・マルチスレッド管理

Codex App の最大の特徴は、複数のプロジェクトとスレッドを1つのウィンドウで管理できることです。サイドバーにはプロジェクトが一覧表示され、各プロジェクトの下に複数のスレッド（会話）が並びます。エージェントは各スレッドで独立して動作するため、「バグ修正」「新機能の実装」「リファクタリング」など複数のタスクを同時に進行させることが可能です。

スレッドを開始する際には、以下の3つのモードから選択できます。

モード 動作 用途
Local 現在のプロジェクトディレクトリで直接作業 日常的な開発タスク、すぐにコードを反映したい場合
Worktree Git Worktree を作成し、分離された環境で作業 実験的な変更、並列タスク、メインの作業を邪魔しない場合
Cloud 設定済みのクラウド環境でリモート実行 ローカルリソースを使いたくない場合、重い処理

Git Worktree ネイティブサポート

Worktree は Codex App の中核的な機能のひとつです。Git の Worktree 機能を活用し、同じリポジトリの複数の独立したチェックアウトを作成して、複数のエージェントが互いに干渉することなく並列で作業できるようにします。

各 Worktree は $CODEX_HOME/worktrees（デフォルトでは ~/.codex/worktrees）配下に作成され、デタッチド HEAD 状態で開始されます。これにより、ブランチを汚さずに作業を進められます。作業が完了したら「Create branch here」ボタンでブランチを作成し、コミット・プッシュ・プルリクエストの作成まで App 内で完結できます。

また、Handoff 機能を使えば、Worktree で作成したスレッドを Local に移動したり、その逆に Local のスレッドを Worktree に退避させたりすることが可能です。Codex が Git 操作を安全に処理してくれるため、手動でのブランチ管理の煩雑さから解放されます。

統合差分レビュー（Diff Pane）

App には差分レビューペインが組み込まれており、エージェントが行った変更を Git diff 形式で確認できます。変更の確認にとどまらず、差分にインラインコメントを追加して Codex にフィードバックしたり、特定のチャンクやファイル単位でステージ/リバートしたり、コミット・プッシュ・PR 作成までアプリ内で完結させることが可能です。

差分表示は「すべての変更」「ステージ済み」「最後のターンの変更のみ」などのフィルターで切り替えられるため、エージェントの変更だけを効率的にレビューできます。

統合ターミナル

各スレッドにはビルトインのターミナルが付属しており、⌘ + J で表示を切り替えられます。ターミナルは現在のプロジェクトまたは Worktree のディレクトリにスコープされているため、git status、npm test、npm start などのコマンドをアプリを離れずに実行できます。変更の検証、開発サーバーの起動、スクリプトの実行などに活用できます。

Skills（スキル）

Skills は Codex の機能をコード生成以外に拡張するための仕組みです。指示書、リソース、スクリプトをバンドルしたパッケージで、Codex がツールに接続し、ワークフローを実行し、チームの好みに応じてタスクを完了できるようにします。

App にはスキルの作成・管理用の専用インターフェースがあり、作成したスキルは App、CLI、IDE 拡張のすべてで利用可能です。また、リポジトリにチェックインすることでチーム全体で共有できます。

OpenAI が公式に提供しているスキルの例を以下に示します。

スキル 機能
Figma Implement Design Figma からデザインコンテキスト・アセット・スクリーンショットを取得し、1:1 のビジュアル忠実度で UI コードに変換
Linear バグのトリアージ、リリース追跡、チームのワークロード管理
Cloudflare / Netlify / Vercel Deploy Web アプリを各種クラウドホストにデプロイ
Image Generation GPT Image を使用して画像を生成・編集（Web サイト、UI モックアップ、ゲームアセット等）
OpenAI Docs OpenAI API の最新ドキュメントを参照しながら開発
PDF / Spreadsheet / Doc PDF、スプレッドシート、Word ファイルの読み取り・作成・編集

スキルの呼び出しはコンポーザーで $skill-name と入力するか、スラッシュコマンド一覧から選択して行います。

Automations（自動化）

Automations は、定期的なスケジュールで Codex にバックグラウンドタスクを実行させる機能です。指示文とオプションのスキルを組み合わせ、定義したスケジュールに基づいて自動的に実行されます。完了した結果はトリアージキュー（受信トレイ）に表示され、確認が必要なものだけをレビューできます。

Git リポジトリでは各 Automation が専用のバックグラウンド Worktree 上で動作するため、メインのチェックアウトに影響を与えません。活用例としては、日次のイシュートリアージ、CI 失敗の検出とサマリー生成、リリースブリーフの自動作成、バグの定期チェック、新しいスキルの自動生成などがあります。

注意: Automations はローカルの Codex App 上で実行されるため、App が起動していて、対象プロジェクトがディスク上に存在している必要があります。

IDE 拡張との同期

VS Code、Cursor、Windsurf などの IDE にインストールした Codex IDE Extension と App は、同じプロジェクトで作業している場合に自動的に同期します。同期すると、App のコンポーザーに「IDE context」オプションが表示され、IDE で現在表示しているファイルをコンテキストとして参照できます。また、App で実行中のスレッドを IDE Extension 側から確認したり、その逆も可能です。

MCP（Model Context Protocol）サポート

Codex App は MCP をサポートしており、外部サービスへの接続を追加できます。App、CLI、IDE Extension で MCP 設定を共有しているため、一度設定すればすべての環境で利用可能です。App の設定画面からおすすめのサーバーを有効化したり、新しいサーバーを追加したりできます。

音声入力・ポップアウトウィンドウ・その他の機能

コンポーザーが表示されている状態で Ctrl + M を長押しすると音声入力が起動し、話した内容が自動的にテキストに変換されます。変換後のテキストを編集してから送信することもできます。

アクティブなスレッドを別ウィンドウにポップアウトして、ブラウザやエディタの近くに配置することも可能です。フロントエンド開発で、ブラウザプレビューを見ながら Codex と対話するのに最適です。「常に最前面に表示」オプションもあります。

その他、画像のドラッグ&ドロップによるコンテキスト追加、タスク完了時のデスクトップ通知、エージェント実行中のスリープ防止機能、パーソナリティの切り替え（/personality コマンド：簡潔な実務的スタイルまたは会話的で共感的なスタイル）なども搭載されています。

Section 05 — Usage Guide

使い方ガイド（起動〜基本操作〜タスク実行の流れ）

Step by Step: 最初のタスクを実行する

  1. App を起動してプロジェクトを追加する

    Codex App を起動し、サイドバーの「Add new project」ボタン（または ⌘ + O）をクリックして、作業したいプロジェクトフォルダを選択します。Git リポジトリのルートディレクトリを選ぶのが一般的です。モノレポの場合は、個別のパッケージやアプリごとに分けてプロジェクトを追加することが推奨されます。

  2. 新しいスレッドを作成する

    ⌘ + N で新しいスレッドを作成します。コンポーザーの下部で「Local」「Worktree」「Cloud」からモードを選択します。初心者はまず「Local」を選びましょう。

  3. タスクを入力して送信する

    コンポーザーにタスクの説明を入力します。例えば「このプロジェクトの README を日本語に翻訳して」「ユニットテストを追加して」「このバグを修正して」などです。Enter キーで送信すると、エージェントが作業を開始します。

  4. エージェントの作業を監視する

    エージェントはファイルの読み取り、コードの編集、コマンドの実行などを行います。承認が必要な操作（ネットワークアクセスやサンドボックス外への書き込みなど）が発生した場合、確認ダイアログが表示されます。「Approve once」（一度だけ許可）や「Approve for this session」（セッション中許可）などのオプションから選択します。

  5. 差分を確認してコミットする

    エージェントの作業が完了したら、⌘ + ⌥ + B で差分パネルを表示し、変更内容を確認します。問題がなければステージしてコミット、必要に応じてプッシュや PR 作成まで App 内で行えます。

ローカル環境設定（Local Environments）

ローカル環境設定を使うと、Worktree 作成時のセットアップスクリプトやプロジェクトでよく使うアクションを定義できます。設定は App の設定画面から行い、プロジェクトルートの .codex フォルダ内に保存されます。Git リポジトリにチェックインすればチームで共有できます。

セットアップスクリプトの例（TypeScript プロジェクトの場合）:

npm install npm run build

アクションの例（開発サーバーの起動）:

npm start

アクションを定義すると、App のトップバーにショートカットボタンとして表示され、ワンクリックで実行できるようになります。

モデルの切り替え方法

Codex App 内でモデルを変更するには、スレッドのコンポーザー付近にあるモデル選択メニューを使用します。CLI では /model コマンドを使ってアクティブなスレッドのモデルを変更するか、起動時に codex -m gpt-5.3-codex のように指定します。

デフォルトモデルは ~/.codex/config.toml で設定できます。

model = "gpt-5.3-codex"

Cloud タスクのモデルは現時点では変更できません（自動的に推奨モデルが使用されます）。

サードパーティモデルの利用

Codex は Chat Completions API または Responses API をサポートする任意のモデルやプロバイダーを指定することも可能です。ただし、Chat Completions API のサポートは非推奨となっており、将来のリリースで削除される予定です。最高のパフォーマンスを得るためには、上記リストのモデルの使用が推奨されます。

Section 08 — Security & Privacy

セキュリティ・プライバシー

ローカル実行のセキュリティモデル

Codex App はオープンソースのシステムレベルサンドボックスを使用しており、CLI と同じセキュリティアーキテクチャを採用しています。セキュリティは「サンドボックスモード」と「承認ポリシー」の2層で構成されています。

サンドボックスモード

macOS では Seatbelt ポリシーが使用され、sandbox-exec を介してコマンドが実行されます。デフォルトでは、エージェントは作業中のフォルダまたはブランチ内のファイル編集と、キャッシュされたWeb検索のみに制限されます。ネットワークアクセスはデフォルトで無効です。

承認ポリシー

エージェントがサンドボックスの範囲外のアクション（ネットワークアクセス、ワークスペース外への書き込み、信頼されていないコマンドの実行など）を行おうとした場合、ユーザーに承認を求めるダイアログが表示されます。承認のスコープは「一度だけ」「このセッション中」などから選択できます。

プリセット フラグ（CLI 相当） 動作
Auto（デフォルト） --full-auto ワークスペース内のファイル読み書き・コマンド実行を自動実行。外部への書き込みやネットワークアクセスは承認が必要。
Read-only --sandbox read-only ファイルの読み取りのみ。編集・コマンド・ネットワークはすべて承認が必要。
Untrusted --ask-for-approval untrusted ファイルの読み書きは自動。信頼されていないコマンドの実行時に承認を要求。
Full Access（危険） --yolo サンドボックスなし・承認なし。非推奨。

保護されるパスとデータ

ワークスペース書き込みモード（デフォルト）であっても、以下のパスは読み取り専用として保護されます。

  • <workspace>/.git — Git メタデータディレクトリ
  • <workspace>/.agents — エージェント設定ディレクトリ
  • <workspace>/.codex — Codex 設定ディレクトリ

また、macOS のセキュリティ機能により、Music、Downloads、Desktop などのシステムディレクトリにアクセスする際には macOS 標準の許可ダイアログが表示されます。

Web 検索とプロンプトインジェクション対策

Codex は Web 検索ツールを標準搭載していますが、デフォルトではキャッシュモード（OpenAI が管理するインデックスからの事前索引済み結果）が使用されます。これにより、ライブ Web コンテンツからのプロンプトインジェクションリスクを軽減しています。ライブ検索が必要な場合は設定で web_search = "live" に変更できますが、その場合は追加のリスクがある点を理解した上で使用してください。

テレメトリ（OTel）と監査

Codex は OpenTelemetry（OTel）によるオプトインモニタリングをサポートしています。デフォルトではテレメトリ送信は無効で、config.toml の [otel] セクションで明示的に有効化する必要があります。有効化すると、会話、API リクエスト、ツール承認の判断、ツールの実行結果などの構造化イベントが記録されます。Enterprise プランではコンプライアンス API を通じた監査ログや使用状況の監視が可能です。

プライバシーに関する注意: log_user_prompt = false（デフォルト）を維持し、プロンプト内容の記録はポリシーが明示的に許可する場合のみ有効化してください。プロンプトにはソースコードや機密データが含まれる可能性があります。

Enterprise 向けセキュリティ機能

Enterprise プランでは、管理者が SAML SSO、MFA、SCIM、暗号化キー管理（EKM）、ドメイン検証、RBAC（ロールベースアクセス制御）、データ保持・データ居住地ポリシーなどを設定できます。管理者による管理設定（requirements.toml）を使って、ワークスペース全体のサンドボックスモードや承認ポリシーを強制することも可能です。また、Business プラン以上ではデフォルトでビジネスデータがモデルの学習に使用されません。

Section 09 — Use Cases

ユースケース集（Mac 開発者向け活用シーン）

バグ修正の並列処理

複数のバグレポートに対して、それぞれ個別の Worktree スレッドを作成し、エージェントに並行して修正を依頼します。各修正は独立した環境で行われるため、互いに干渉しません。修正が完了したら差分をレビューし、問題なければブランチを作成して PR を提出します。1人の開発者が同時に5つ以上のバグ修正を進行管理できるようになります。

Figma デザインの実装

Figma Implement Design スキルを使い、Figma のデザインを UI コードに変換します。デザイナーから受け取ったモックアップの URL を指定するだけで、コンポーネントの構造、スタイリング、レスポンシブ対応まで自動的に実装されます。ポップアウトウィンドウでブラウザプレビューを並べながら微調整も可能です。

日次の自動レポート

Automations を設定して、毎朝リポジトリの過去24時間のコミットを分析し、ワークストリームごとのエグゼクティブブリーフを自動生成します。PR リンクやレビューコメントも含めた Markdown レポートが受信トレイに届きます。チームリーダーやプロジェクトマネージャーにとって強力な情報集約ツールです。

テスト自動生成

「このプロジェクトのすべての関数にユニットテストを追加して」と指示するだけで、エージェントがコードベースを分析し、テストファイルを自動生成します。生成されたテストは統合ターミナルですぐに実行して検証でき、不足があればフィードバックして追加させられます。

CI 失敗の自動トリアージ

Automations とスキルを組み合わせ、CI/CD パイプラインの失敗を定期的にチェックし、失敗原因の分析と修正案の提案を自動化します。OpenAI 社内では、この種の自動化を実際に運用しているとのことです。

リファクタリング

大規模なリファクタリングタスクを Worktree で安全に実行します。「このモジュールを TypeScript に移行して」「このクラスの責務を分離して」といった指示を複数のスレッドに分割し、並列で進行させることが可能です。7百万トークン以上を使用した長時間タスクも実行されています。

Web アプリのデプロイ

Cloudflare、Netlify、Vercel、Render などのデプロイスキルを使い、開発したWebアプリのデプロイまでをCodex内で完結させます。デプロイの設定、ビルド、公開までをエージェントが処理します。

ゲーム開発

OpenAI のブログでは、Image Generation スキルと Web Game Development スキルを組み合わせ、レーシングゲーム（複数のレーサー、8つのマップ、アイテム機能付き）を1つの初期プロンプトだけで構築した事例が紹介されています。エージェントがデザイナー、開発者、QA テスターの役割を担い、7百万トークン以上を使用して完成させました。

Section 10 — Limitations

制限事項・既知の問題・注意点

プラットフォームの制限

  • Intel Mac 非対応: Codex App は Apple Silicon 専用です。M1 以降のチップが必要で、Intel Mac では動作しません。Intel Mac ユーザーは CLI 版または Cloud 版を利用してください。
  • Linux 非対応（App として）: デスクトップアプリは macOS と Windows のみです。Linux では CLI 版を使用します。ただし、コミュニティによる非公式な Linux 上での DMG 実行の報告もあります。

Worktree に関する注意点

  • Git リポジトリが必要: Worktree 機能は Git リポジトリ内のプロジェクトでのみ利用可能です。
  • ブランチの排他制御: Git の制約により、同一ブランチを複数の Worktree で同時にチェックアウトすることはできません。Worktree でブランチを作成した場合、そのブランチは Local でチェックアウトできなくなります。Handoff 機能で安全に移動してください。
  • ディスク容量: 各 Worktree はリポジトリのファイル、依存関係、ビルドキャッシュの独立したコピーを持つため、ディスク容量を大量に消費する可能性があります。デフォルトで最新15個の Worktree が保持され、古いものは自動削除されます。
  • .gitignore のファイル: Handoff 時に .gitignore に記載されたファイルは移動されません。
  • 依存関係のセットアップ: Worktree は Git にコミットされたファイルのみを継承するため、node_modules などの依存関係は含まれません。Local Environment のセットアップスクリプトで npm install などを定義する必要があります。

Automations に関する注意点

  • App の起動が必要: Automations はローカルで実行されるため、Codex App が起動していなければ動作しません（クラウドベースのトリガーは今後サポート予定）。
  • Worktree の蓄積: 頻繁な Automation は大量の Worktree を生成します。不要な実行結果はアーカイブし、必要なもの以外はピン留めしないようにしましょう。
  • サンドボックス設定: Automations はデフォルトのサンドボックス設定で実行されます。Read-only モードではファイル変更・ネットワークアクセスが失敗するため、適切なモードに設定する必要があります。

その他の既知の問題

  • レビューパネルの表示: Git リポジトリ内のプロジェクトでは、Codex が編集していない変更もレビューパネルに表示されることがあります。「Last turn changes」ビューに切り替えることで、最新のターンの変更のみを表示できます。
  • Apple Music へのアクセス要求: タスクによってはファイルシステムの探索が必要になり、macOS が Music、Downloads、Desktop などのフォルダへのアクセス許可を求めるダイアログを表示することがあります。
  • CLI と App のバージョン差: App と CLI は同じ Codex エージェントを使用していますが、内部バージョンが異なる場合があり、一部の実験的機能が CLI でのみ先行提供されることがあります。
  • API キーでの機能制限: API キーでサインインした場合、Cloud スレッド、GitHub コードレビュー、Slack 連携などのクラウドベース機能は利用できません。
  • Cloud タスクのモデル固定: Cloud タスクで使用されるモデルは現時点では変更できません。
  • スレッドのスタック: まれにスレッドが停止することがあります。その場合は、承認待ちがないか確認し、ターミナルで基本コマンドを実行してみてください。解決しない場合は新しいスレッドを作成します。

ログとトラブルシューティング

問題が発生した場合、以下の場所にログが保存されています。

ログの種類 場所
App ログ（macOS） ~/Library/Logs/com.openai.codex/YYYY/MM/DD
セッション記録 ~/.codex/sessions
アーカイブ済みセッション ~/.codex/archived_sessions

フィードバックを送信するには、コンポーザーに /feedback と入力します。既存のセッションを共有することもできます。GitHub の openai/codex リポジトリで既知のイシューを確認したり、新しいイシューを報告したりすることもできます。

どれを選ぶべきか？

3つの Codex はそれぞれ補完的な役割を持っており、多くの開発者は複数を併用しています。設定やセッション履歴は共有されるため、状況に応じて最適なインターフェースを使い分けることが推奨されます。

Mac 開発者で複数のタスクを日常的に並列処理するなら、Codex App がメインツールとして最適です。ターミナルでの素早い操作が必要なシーンでは CLI を、外出先やローカル環境がない状況では Cloud を活用するのが良いでしょう。

Section 12 — Tips & Best Practices

Tips・ベストプラクティス

プロンプトの書き方

Codex は明確で具体的な指示に最もよく反応します。OpenAI の公式ドキュメントでは「Codex をチームメイトのように扱い、明示的なコンテキストと明確な『完了の定義』を与える」ことが推奨されています。

  • タスクを具体的に記述する: 「コードを改善して」ではなく「この関数のエラーハンドリングを追加し、try-catch ブロックでラップして、エラー時にはログ出力する」のように明確に書きます。
  • コンテキストを適切に提供する: 関連するファイルパスやディレクトリを明示し、プロジェクトの構造を理解させます。画像のドラッグ&ドロップやスキルの活用も効果的です。
  • 完了条件を明示する: 「テストが通ること」「リントエラーがないこと」「既存の API と互換性があること」など、完了と見なす条件を明記します。

AGENTS.md を活用する

AGENTS.md ファイルはプロジェクトのルートやサブディレクトリに配置する設定ファイルで、Codex に対してプロジェクト固有の指示やコーディング規約を伝えるために使います。大規模プロジェクトでは、ルートに全体的な指示を書き、サブディレクトリにはそのモジュール固有の指示をネストさせることで、コンテキスト量を制御できます。

使用量を節約するコツ: AGENTS.md のサイズが大きすぎると、各メッセージで消費されるコンテキスト量が増加します。必要最小限の情報に絞り、詳細はサブディレクトリのファイルに分散させましょう。

モデルの使い分け

日常的なルーティンタスク（軽微な修正、フォーマット変更、簡単なテスト追加など）には GPT-5.1-Codex-Mini を使用し、使用量上限をおよそ4倍に延ばしましょう。複雑なアーキテクチャ変更やマルチファイルのリファクタリングなど、高度な推論が必要なタスクには GPT-5.3-Codex を使います。Pro ユーザーはリアルタイムのペアプログラミングに GPT-5.3-Codex-Spark を活用できます。

Git ワークフローのベストプラクティス

  • フィーチャーブランチで作業する: git status がクリーンな状態で Codex にタスクを委任すると、変更の分離とリバートが容易になります。
  • こまめにコミットする: 小さなインクリメントでコミットすることで、問題が発生した場合のロールバックが簡単になります。
  • Codex の変更を PR として扱う: 対象を絞った検証を実行し、差分をレビューし、コミットメッセージに判断の根拠を記録します。
  • Worktree を活用する: 実験的な変更は Worktree で行い、メインの作業環境を汚さないようにしましょう。

MCP サーバーの管理

追加する MCP サーバーごとにメッセージのコンテキスト量が増加し、使用量の消費が早くなります。使用していない MCP サーバーは無効化しておき、必要な時だけ有効にしましょう。

Automations を段階的に導入する

新しい Automation を設定する前に、まず通常のスレッドでプロンプトを手動テストし、期待通りの結果が得られることを確認してください。スケジュール実行を開始したら、最初の数回の出力を注意深くレビューし、プロンプトや頻度を調整します。不要になった Automation の実行結果はアーカイブして、Worktree の蓄積を防ぎましょう。

パフォーマンスの最適化

  • スリープ防止を活用する: 長時間タスクを実行する場合、設定で「Prevent sleep while running」を有効にして、Mac のスリープによるタスクの中断を防ぎます。
  • 通知を活用する: App がバックグラウンドにある時にタスク完了や承認要求の通知を受け取れるようにしておくと、他の作業に集中しながら効率的にエージェントを管理できます。
  • ポップアウトウィンドウを活用する: フロントエンド開発では、スレッドをポップアウトしてブラウザの横に配置し、「常に最前面」にしておくとワークフローが快適になります。

スキルを自動的に改善する

Automations の高度な活用として、過去のセッションを分析して既存スキルの改善や新スキルの自動生成を行うことができます。以下のようなプロンプトで Automation を設定します。

過去24時間の ~/.codex/sessions ファイルをスキャンし、 特定のスキルの使用で問題があった場合はスキルを更新してください。 頻繁に行っている作業で、スキルとして保存すべきものがあれば 新しいスキルを作成してください。 更新が必要な場合のみ変更してください。

このように、Codex 自身にスキルのメンテナンスを委任することで、チームのワークフローを継続的に改善できます。
