LLMは長いリスト（数百〜数千行）から正しいレコードを見つけられるか

調査日: 2026-03-15 ｜ 対象: Claude 3.5/4系, GPT-4o等の最新モデル ｜ 学術論文・ベンチマーク結果に基づく

1. "Lost in the Middle" 問題の現状

元論文の知見 (Liu et al., 2023)

LLMは長い文脈の先頭と末尾に置かれた情報の検索精度が高く、中間部分の情報を見落としやすい。原因はRotary Position Embedding (RoPE) の長距離減衰効果にある。

最新モデルでの改善状況

Databricksの長文脈RAGベンチマーク（2024-2025）によれば:

  • GPT-4o、Claude 3.5 Sonnet、GPT-4o-miniは、文脈長が増加しても性能劣化がほとんど見られない
  • 旧モデル（GPT-4-turbo、Claude 3 Sonnet）では16Kトークン付近で飽和していた問題が大幅に改善
  • GPT-4oとClaude 3.5 Sonnetは32Kトークンまで一貫してRAG性能が向上し続ける
  • o1-preview / o1-miniは全モデル中最高精度を記録

結論: 最新モデルではLost in the Middleは実質的にほぼ解消

明示的キーワードマッチング vs 暗黙的推論

タスク種別難易度説明
明示的キーワードマッチ
（TSV内の特定文字列検索）		容易Needle in a Haystackテストと本質的に同じタスク。最新モデルは99%+の精度
暗黙的推論
（文脈から判断）		やや困難複数の情報を組み合わせる必要があり、文脈長が増すと精度低下。Sequential-NIAHでは最良のGemini 1.5でも63%程度

10Kトークンのリストの場合: 明示的キーワード検索であれば、最新モデルでは実用上問題にならない水準（99%+）。ただし「このリストの中で条件Xかつ条件Yを満たすものを全て列挙せよ」のような複合推論は精度が落ちる。

2. Needle in a Haystack (NIAH) ベンチマーク

主要モデルのNIAH性能

モデルNIAH精度最大検証文脈長備考
Claude 3 Opus99%+200Kトークン30種のランダムneedle/質問ペアで検証。needle挿入の人為性を指摘した事例も
Claude 3.5 Sonnet99%+200Kトークン文脈長増加による劣化がほぼ無い

Gemini 1.5 Pro高精度2Mトークン極長文脈でも一貫した性能を維持

日本語でのNIAH性能（MLNeedle研究, 2024）

多言語NIAHベンチマーク（MLNeedle, arxiv:2408.10151）の知見:

  • 非ラテン言語（日本語・中国語・アラビア語）ではNIAH精度が顕著に低下
  • 英語→非ラテン言語への切替で大幅な性能ドロップが観測
  • 特に中間位置 + 非ラテン言語の組合せが最も精度が低い
  • 8Kトークンを超えると多言語検索精度が著しく低下。32K対応を謳うモデルでも有効性能は限定的
  • 周囲のテキスト（haystack）の言語よりも、needle自体の言語が精度に強く影響

結論: 日本語テキスト内の日本語needleの検索は、英語より明確に不利。10Kトークン程度でも注意が必要

3. 構造化データ（TSV/テーブル）の読解力

テーブルフォーマット別LLM理解度（11形式比較）

Improving Agents (2025) の11フォーマット比較ベンチマーク:

フォーマット精度トークン効率評価
Markdown-KV (key: value形式)60.7%低（CSV比2.7倍）最高精度
HTML Table~65% (with prompt)低高精度
Markdown Table中程度中バランス型
CSV / TSV~44%高（最小トークン）精度低い
JSONL低中非推奨
Pipe-Delimited低低非効率

重要: TSV/CSVはトークン効率は良いが、LLMの理解精度は最も低い部類。HTML TableやMarkdown-KVが精度面で優位。

列フィルタリング・キーワード抽出の精度

  • LLMは本質的に線形テキスト処理を行うため、行×列の空間的関係の理解が弱い
  • 特定列のフィルタリング: 可能だが完全ではない。ヘッダーの認識と各行での対応関係を正しく追跡する必要がある
  • キーワード完全一致検索: 明示的な文字列であれば高精度（NIAHと同じメカニズム）
  • 意味的フィルタリング（「税務関連の行だけ抽出」等）: 精度が大幅に低下

行数増加と精度の関係

  • LLMは最初の約40行を処理した後に停止する傾向がある（Dust.tt報告）
  • U字型位置バイアス: リストの先頭と末尾のレコードは見つけやすく、中間部分は見落としやすい
  • ヘッダーの繰り返し（100行ごとにヘッダーを再挿入）で改善効果あり
  • 数百行を超えると、全件正確な抽出は困難。バッチ処理（1回に数十行ずつ）が推奨

4. 実用的な上限の目安

タスク別の推奨行数上限

タスク推奨上限根拠
特定キーワードの存在確認
（「Xという文字列を含む行はあるか？」）		 数百行（~500行） NIAHテストと同等。最新モデルは200Kトークンまで99%+。10Kトークン（約500行TSV）なら問題なし
特定キーワードを含む全行の列挙
（「Xを含む行を全て抜き出せ」）		 100〜200行 全件漏れなく列挙するには注意力の持続が必要。中間部分の見落としリスクあり
条件付きフィルタリング
（「A列がXかつB列 > Y のもの」）		 50〜100行 複合条件は推論負荷が高い。テーブル構造の空間理解が必要
集計・カウント
（「条件Xを満たす行は何件か」）		 30〜50行 LLMのカウント能力は根本的に弱い。正確性を求めるならコード実行が必須
日本語テキストでの検索 英語の6〜7割程度に減算 MLNeedle研究で非ラテン言語の性能低下が確認済み。トークン化効率の差も影響

5. 精度を上げるための実践的テクニック

推奨アプローチ

  1. フォーマット選択: TSVよりMarkdown Table またはHTMLテーブルを使う（精度+16ポイント程度の差）
  2. ヘッダー繰返し: 100行ごとにカラムヘッダーを再挿入する
  3. バッチ分割: 数百行以上のデータは50〜100行ずつに分割して処理
  4. 重要情報の配置: 検索対象が分かっている場合、リストの先頭か末尾に配置
  5. 明示的な指示: 「全ての行を1行ずつチェックして、キーワードXを含む行を列挙せよ」のように手順を明示
  6. コード生成の活用: 大量データのフィルタリングは、LLMにPythonコードを書かせて実行する方が確実
  7. ハイブリッド検索: 前処理としてgrepや正規表現で候補を絞り、LLMには意味的判断のみを委ねる

6. 総合判定

10Kトークン（約500行TSV）でのキーワード検索は実用可能か？

回答: 条件付きでYes。

  • 単一キーワードの存在確認: 最新モデル（Claude 3.5+, GPT-4o）であれば、10Kトークン程度ならほぼ確実に見つけられる（99%+）
  • 全件列挙: 該当行が少数（5件以下）なら高精度。該当行が多い（10件+）場合は漏れのリスクあり
  • 日本語の場合: 英語より精度が落ちるため、300行程度を上限の目安とする方が安全
  • TSV形式のままでは精度が低い: 可能であればMarkdown TableかHTML Tableに変換して渡すと精度向上
  • 確実性が必要な場合: LLMに直接検索させるのではなく、grepやPythonコードを書かせて実行する方が圧倒的に信頼性が高い
