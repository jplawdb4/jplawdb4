Section 01 — Overview

Codex Windows版とは何か

OpenAI Codex（コーデックス）は、OpenAI が提供する AI コーディングエージェントです。自然言語の指示をもとにコードの読み書き・実行・レビュー・デバッグまで一貫して行い、開発者の生産性を大幅に引き上げることを目的としています。2025年4月に CLI ツールとして初登場し、2025年5月にはChatGPT 内のクラウド版（旧称 Codex）が、そして2026年2月にはmacOS デスクトップアプリがリリースされました。

そして 2026年3月4日（日本時間3月5日）、ついに Windows 版ネイティブデスクトップアプリ が正式リリースされました。Microsoft Store からダウンロードでき、PowerShell 上でネイティブに動作する Windows サンドボックスを備えた、Windows 開発者待望の公式アプリケーションです。

Codex の中心にある思想は「すべてはコードで制御される」という前提です。コードの推論能力が高いモデルであればあるほど、技術的な作業だけでなく、ドキュメント作成、データ分析、テスト自動化といった幅広い知的作業を代行できるようになります。Codex アプリは、そうした AI エージェントへの指示・監督・協働を効率的に行うための「コマンドセンター」として設計されました。

CLI版・クラウド版・Mac版との違い

Codex には現在 4 つの利用形態があります。CLI版はターミナル上で動作する軽量なコーディングエージェントで、npm でインストールしてコマンドラインから直接操作します。クラウド版は ChatGPT のウェブインターフェース内で動作し、OpenAI が管理するクラウドコンテナ上でコードを実行します。ローカルにリポジトリがなくても GitHub 連携でコードを扱えるのが特徴です。Mac版デスクトップアプリは 2026年2月2日にリリースされた macOS 専用のネイティブアプリで、Apple Silicon に最適化されています。

今回の Windows版デスクトップアプリは、Mac版と同等のコア機能（マルチスレッド並列実行、ワークツリー、Skills、Automations、Git 統合、内蔵ターミナル）を備えつつ、Windows 固有の環境に最適化されています。PowerShell をデフォルトシェルとし、WSL（Windows Subsystem for Linux）を使わずにネイティブ Windows サンドボックスで安全に動作します。もちろん、WSL 環境を好むユーザーはエージェントを WSL モードに切り替えることも可能です。

Section 02 — System Requirements

システム要件・対応Windowsバージョン

Codex Windows版は Microsoft Store 経由で配布されるため、Microsoft Store に対応した Windows 10 以降のバージョンが必要です。推奨環境は Windows 11（最新アップデート適用済み）です。Codex はローカルで AI モデルを実行するわけではなく、OpenAI のクラウド API にリクエストを送信して結果を受け取る設計であるため、GPU は不要です。ただし、エージェントが生成したコードのビルドや実行にはそれなりのCPU・メモリリソースが望ましいです。

Codex はコード生成・実行をローカルで行いますが、AI の推論自体は OpenAI のクラウドで処理されます。そのため、ローカルに強力な GPU は不要で、安定したインターネット接続が最も重要な要件です。
Section 03 — Installation

インストール方法

Windows版 Codex アプリは Microsoft Store から無料でダウンロードできます。エンタープライズ環境では、管理者が Microsoft Store のアプリ配布機能を通じて組織内に一括展開することも可能です。以下の手順で、数分でセットアップが完了します。

Microsoft Store からダウンロード

Microsoft Store を開き、「Codex」で検索するか、直接リンク（https://apps.microsoft.com/detail/9plm9xgg6vks）にアクセスします。「入手」ボタンをクリックすると自動でダウンロード・インストールされます。

Codex を起動してサインイン

インストール後、スタートメニューから「Codex」を起動します。初回起動時にサインイン画面が表示されるので、ChatGPT アカウントまたはOpenAI API キーでログインします。ChatGPT アカウントでログインすると、クラウドスレッドやGitHub連携などのフル機能が利用できます。API キーでのログインは一部機能が制限されます。

プロジェクトフォルダを選択

ログイン後、「Let's build」画面でプロジェクトフォルダを選択します。既存のGitリポジトリを開いても、新しいフォルダを指定しても構いません。WSL ファイルシステム上のフォルダを開く場合は、ファイルダイアログで \\wsl$\ と入力して Linux ディストリビューション内のフォルダを参照できます。

開発ツールをインストール（推奨）

Codex を最大限に活用するために、いくつかの開発ツールを事前にインストールしておくと便利です。内蔵ターミナルまたは PowerShell から以下を実行します。

PowerShell # Git — レビュー・diff・ワークツリーに必須 winget install --id Git.Git # Node.js — エージェントのタスク実行効率化 winget install --id OpenJS.NodeJS.LTS # Python — 各種スクリプト実行 winget install --id Python.Python.3.14 # .NET SDK — Windows ネイティブアプリ開発時 winget install --id Microsoft.DotNet.SDK.10 # GitHub CLI — PR作成等のGitHub連携 winget install --id GitHub.cli

最初のメッセージを送信

プロジェクトを選択したら、スレッドモードを「Local」に設定し、テキスト入力欄に日本語または英語で指示を入力します。たとえば「このプロジェクトの構造を説明して」「READMEを日本語で作成して」「テストが失敗している原因を調べて」など、自然言語でなんでも聞けます。

アップデート方法
Microsoft Store を開き、「ダウンロード」セクションで「更新プログラムの確認」をクリックすると最新版に自動アップデートされます。Codex は頻繁にアップデートされるため（2026年2〜3月は日次レベル）、こまめな更新を推奨します。
Section 04 — Features

主な機能と特徴

Codex デスクトップアプリは、単なる「チャットでコード生成」ツールではありません。複数のAIエージェントを並行して管理し、長時間にわたるタスクを監督・協働するための「エージェントのコマンドセンター」です。ここでは Windows 版に搭載された主要機能を解説します。

🧵

マルチスレッド並列実行

複数のエージェントスレッドをプロジェクト単位で管理し、タスク間をシームレスに切り替えられます。コンテキストを失わずに複数の作業を同時進行できます。

🌳

Git ワークツリー

各エージェントが独立した Git ワークツリーで作業するため、同じリポジトリでの並行作業が競合なく行えます。メインブランチに影響を与えずに実験できます。

🔧

Skills（スキル）

指示・リソース・スクリプトをバンドルしたスキルで Codex の能力を拡張。Figma デザイン実装、画像生成、クラウドデプロイ、ドキュメント作成など多数のスキルが用意されています。

⚡

Automations（自動化）

定期スケジュールで Codex をバックグラウンド実行。CI障害の要約、イシュートリアージ、日次レポート生成など反復タスクを自動化できます。

📝

Git 統合 & Diff レビュー

変更差分をアプリ内でレビュー、インラインコメント、チャンクの stage / revert、コミット、プッシュ、PR 作成までアプリを離れずに完結。

💻

内蔵ターミナル

各スレッドごとに専用ターミナルを内蔵。テスト実行、dev サーバー起動、Git 操作などをアプリ内で直接行えます。PowerShell、CMD、Git Bash、WSL から選択可能。

🔌

MCP サポート

Model Context Protocol（MCP）で外部サービスと接続。CLI や IDE 拡張と MCP 設定を共有し、設定済みのサーバーをそのまま利用できます。

🖥️

Windows ネイティブ実行

PowerShell + ネイティブ Windows サンドボックスでWSL不要で動作。WSL/VM なしでもセキュアなコード実行が可能です。

🔗

IDE 連携同期

VS Code の Codex IDE 拡張と自動同期。エディタで開いているファイルのコンテキストをアプリ側が追跡し、双方向でスレッドを共有します。

🎤

音声入力

Ctrl+M を押しながら話すと音声がテキストに変換されます。手がふさがっているときもプロンプトを入力可能。

🪟

ポップアウトウィンドウ

会話スレッドを別ウィンドウに切り離し可能。Always-on-Top にも対応し、ブラウザやエディタと並べて使えます。

🖼️

画像入力 & 生成

画像のドラッグ＆ドロップでコンテキスト追加。スクリーンショット連携や GPT Image を使った画像生成スキルにも対応。

Section 05 — Usage Guide

使い方ガイド

基本的なワークフロー

Codex アプリを起動すると、左サイドバーにプロジェクト一覧、中央にスレッドエリア、右側に diff パネルとターミナルが配置されたインターフェースが表示されます。CLI や IDE 拡張の利用履歴がある場合は、過去のプロジェクトやセッションも自動的に引き継がれます。

プロジェクトを開く

サイドバーの「＋」ボタンまたは Ctrl+O でプロジェクトフォルダを選択します。Git リポジトリを開くと、ワークツリーやブランチ管理機能がフルに使えます。複数プロジェクトを同時に登録して切り替えることも可能です。

スレッドモードを選択

新しいスレッドを開始する際、3つのモードから選択します。Local は現在のプロジェクトディレクトリで直接作業します。Worktree は Git ワークツリーを自動作成し、変更を本体から隔離して作業します。Cloud は OpenAI のクラウド環境でリモート実行します。通常は Local または Worktree を使用します。

自然言語で指示する

入力欄に日本語で指示を入力し、送信します。Codex はプロジェクトのコードを読み、必要に応じてファイルの編集やコマンド実行を行います。

プロンプト例 # バグ修正 "src/api/auth.ts の認証ロジックで、トークン期限切れ時に500エラーが返る問題を修正して" # 機能追加 "ユーザーダッシュボードにCSVエクスポート機能を追加して。ボタンのデザインは既存のUIに合わせて" # コードベースの理解 "このプロジェクトのアーキテクチャを図で説明して。主要なモジュール間の依存関係を示して" # テスト作成 "src/utils/ ディレクトリのすべての関数に対するユニットテストを作成して"

承認 (Approval) を管理する

Codex がファイル書き込みやコマンド実行を行う際、サンドボックスの設定に応じて承認を求めるダイアログが表示されます。「Approve once（今回だけ許可）」や「Approve for this session（このセッション中許可）」を選んで、安全性と利便性のバランスを取ります。

変更をレビューする

エージェントが作業を完了すると、diff パネルに変更差分が表示されます。インラインでコメントを追加したり、特定のチャンク（変更塊）を stage / revert したりできます。修正が必要な場合はコメントで指示すると、Codex が修正を加えます。

コミット・プッシュ・PR作成

レビュー完了後、アプリ内から直接 Git コミット、プッシュ、プルリクエスト作成まで行えます。GitHub CLI (gh) がインストールされていれば、PR の作成もワンクリックです。

便利なキーボードショートカット
Ctrl+J — ターミナルの表示/非表示。Ctrl+K — コマンドパレットを開く。Ctrl+M(長押し) — 音声入力。Ctrl+O — プロジェクトを開く。
Section 06 — Models

対応モデル

Codex は OpenAI の最先端モデルを活用してコーディングタスクを実行します。デフォルトでは最新の推奨モデルが自動選択されますが、タスクの性質やコスト・速度の好みに応じてモデルを切り替えることができます。アプリの設定画面やスレッド内の /model コマンドで変更可能です。

モデル名 特徴 アプリ対応 利用プラン
GPT-5.3-Codex 最も高性能なコーディングモデル。フロンティア推論＋専門知識を両立 対応 Plus / Pro / Business / Enterprise
GPT-5.3-Codex-Spark 超高速リアルタイムモデル（~1000 tokens/s）。日常的な編集作業に最適 対応 Pro のみ（研究プレビュー）
GPT-5.2-Codex 実務的なコーディングモデル。GPT-5.3 の前世代 対応 全有料プラン

GPT-5.2 / GPT-5.1 汎用エージェントモデル。コーディング以外のタスクにも対応 対応 全有料プラン
サードパーティモデル Chat Completions / Responses API 対応の任意モデルを指定可能 設定で対応 API キー認証時
モデル設定方法

デフォルトモデルは config.toml ファイルで設定できます。CLI・IDE 拡張・デスクトップアプリでこの設定ファイルを共有するため、一度設定すればすべての Codex 環境に反映されます。

config.toml # デフォルトモデルを指定 model = "gpt-5.3-codex" # コスト効率を重視する場合 # model = "gpt-5.1-codex-mini"

Section 08 — Security & Privacy

セキュリティ・プライバシー

Codex は「セキュリティ・バイ・デザイン」の理念で設計されており、ローカル実行時のセキュリティは 2 つのレイヤーで構成されています。サンドボックスは OS レベルでエージェントがアクセスできるファイルシステムとネットワークを制限し、承認ポリシーはエージェントがアクションを実行する前にユーザーの許可を求めるタイミングを制御します。

🔒

Windows ネイティブ サンドボックス

Windows 版では、Restricted Token アプローチとファイルシステム ACL を使用して書き込み可能なディレクトリを制限します。専用の Sandbox User としてコマンドを実行し、Windows Firewall ルールでネットワークアクセスも制限します。unelevated / elevated の 2 モードから選択可能です。

🌐

ネットワーク制御

デフォルトでネットワークアクセスはオフ。Web 検索はキャッシュ経由で提供され、プロンプトインジェクションのリスクを低減します。ライブ検索やネットワークアクセスは明示的に有効化する必要があります。

📂

保護パス

.git、.agents、.codex ディレクトリは書き込み可能なルートであっても自動的に読み取り専用として保護されます。~/.ssh などの機密ディレクトリへのサンドボックスからの読み取りアクセスもブロックされます。

📊

監査ログ (OTel)

OpenTelemetry によるオプトイン監査ログに対応。API リクエスト、ツール承認判断、コマンド実行結果などを構造化ログとして自社のコレクターに送信可能です。デフォルトではオフです。

サンドボックス設定の組み合わせ
用途 サンドボックスモード 動作
Auto（デフォルト） workspace-write + on-request ワークスペース内の読み書き・コマンド実行は自動。外部アクセスは承認が必要
読み取り専用 read-only + on-request ファイル読み取りと質問応答のみ。編集・実行・ネットワークは承認が必要
CI向け read-only + never 読み取りのみ、承認プロンプトなし
フルアクセス danger-full-access サンドボックスも承認もなし（非推奨）

プライバシーについて — OpenAI API に送信されたデータは、ユーザーが明示的にオプトインしない限り、OpenAI モデルのトレーニングに使用されません（2023年3月1日以降のポリシー）。Business / Enterprise プランでは、デフォルトでビジネスデータがトレーニングに使用されない設定になっています。ローカル実行時のコードはユーザーのマシン上で処理され、プロンプトと応答のみがOpenAI のサーバーと通信します。

config.toml — Windows サンドボックス設定 # Windows ネイティブ サンドボックスモード [windows] sandbox = "elevated" # "unelevated" または "elevated" # ワークスペース書き込みモードでのネットワークアクセス（デフォルト: false） [sandbox_workspace_write] network_access = false # Web検索設定（"cached" | "live" | "disabled"） web_search = "cached" # OTel 監査ログ（オプション） [otel] environment = "prod" exporter = "none" log_user_prompt = false
Section 09 — Windows-Specific

Windows固有の機能・注意点

Windows 版 Codex アプリには、Windows 環境ならではの設定項目や注意点がいくつかあります。ここでは PowerShell 実行ポリシー、WSL 連携、パス問題、エディタ設定などの Windows 固有のトピックを解説します。

PowerShell 実行ポリシーの問題

PowerShell のデフォルト実行ポリシーが Restricted の場合、Codex エージェントや内蔵ターミナルで npm.ps1 cannot be loaded because running scripts is disabled on this system. のようなエラーが発生することがあります。これは Codex がスクリプトを生成して実行する際にもブロックされる原因となります。

PowerShell — 実行ポリシーの変更 # 現在のポリシーを確認 Get-ExecutionPolicy # RemoteSigned に変更（ローカルスクリプトは許可、リモートは署名が必要） Set-ExecutionPolicy -ExecutionPolicy RemoteSigned
WSL との連携

Codex アプリはデフォルトで Windows ネイティブエージェント（PowerShell ベース）で動作しますが、設定画面からエージェントを WSL モードに切り替えることもできます。WSL モードでは Linux のサンドボックス実装（Landlock + seccomp）が使用されます。切り替え後はアプリの再起動が必要です。

WSL プロジェクトを Windows ネイティブエージェントで開く場合は、ファイルダイアログで \\wsl$\ を入力して WSL 内のフォルダを参照できます。ただし、パフォーマンスの観点から、Windows ネイティブエージェント使用時はプロジェクトを Windows ファイルシステム上に置き、WSL からは /mnt/c/... 経由でアクセスする構成が推奨されます。

内蔵ターミナルの選択

アプリ設定で内蔵ターミナルのデフォルトシェルを選択できます。PowerShell、Command Prompt、Git Bash、WSL の 4 つのオプションがあり、インストールされているものが表示されます。エージェントモードとターミナル設定は独立しているため、エージェントを WSL で動かしつつターミナルは PowerShell を使う、といった組み合わせも可能です。変更は新しいターミナルセッションから反映されます。

管理者権限での実行

Codex が昇格された権限（管理者権限）でコマンドを実行する必要がある場合は、Codex アプリ自体を管理者として起動する必要があります。スタートメニューで「Codex」を右クリックし、「管理者として実行」を選択します。エージェントはアプリのパーミッションレベルを継承します。

Windows ローカル環境スクリプト

Codex アプリの「ローカル環境」機能では、プロジェクトごとにセットアップスクリプトやアクション（ワンクリックで実行できるコマンド）を定義できます。npm run dev のようなクロスプラットフォームコマンドなら共有スクリプト1つで済みますが、Windows 固有の処理が必要な場合は Windows 専用のセットアップスクリプトやアクションを作成します。アクションは内蔵ターミナルの環境で実行され、セットアップスクリプトはエージェント環境（WSL またはPowerShell）で実行されます。

パスの注意点
WSL ファイルシステム上のプロジェクトを直接 Windows ネイティブエージェントで扱うと、/mnt/c/ 経由のパスは I/O が遅く、シンボリックリンクやパーミッションの問題が発生しやすくなります。WSL を使う場合はリポジトリを ~/code/ 以下の Linux ネイティブパスに置き、エージェントも WSL モードにするか、Windows ネイティブで使うなら C: ドライブ上にプロジェクトを配置しましょう。
Git 未インストール時の制限
Windows にネイティブの Git がインストールされていない場合、レビューパネル、diff 表示、ワークツリーなどの Git 関連機能が使用できません。winget install Git.Git で必ず事前にインストールしてください。

🐛

バグ修正・デバッグ

エラーログやスタックトレースを貼り付けて「この問題を修正して」と指示するだけ。Codex がコードベースを探索し、原因を特定してパッチを提案します。C#/.NET プロジェクトの例外処理修正にも対応。

🏗️

.NET / C# 開発

ASP.NET Core の API エンドポイント追加、Entity Framework のマイグレーション作成、WPF / WinUI のUIコンポーネント生成など、Windows ネイティブ開発に活躍。.NET SDK との連携で MSBuild プロジェクトもスムーズに操作。

🧪

テスト自動生成

「このディレクトリのすべての関数にユニットテストを作成して」と指示すると、xUnit / NUnit / Jest 等のテストコードを自動生成。既存テストのパターンに合わせて書くことも可能。

📄

レガシーコード移行

🎨

🚀

クラウドデプロイ

📋

日次自動レポート

Automations で毎朝 CI 障害の要約、Sentry エラーのトリアージ、コードベースの変更サマリーを自動生成。レビューキューに結果が届くので確認するだけ。

Section 11 — Limitations

制限事項・既知の問題

Windows 版 Codex アプリは 2026年3月4日にリリースされたばかりであり、今後のアップデートで改善される見込みのある制限事項や既知の問題がいくつか存在します。

Windows版の制限事項

リリース直後のバージョン問題 — 初期に Microsoft Store からダウンロードした一部のユーザーから「This version of the app is no longer supported. Download the latest version here.」というメッセージが報告されています。Microsoft Store の「更新プログラムの確認」から最新版へ更新することで解決します。Codex は非常に頻繁にアップデートされるため、インストール直後に更新確認を行うことを推奨します。

Automations のローカル実行制限 — Automations は現在、ラップトップの電源がオンの状態でのみ実行されます。クラウドベースのトリガーによるバックグラウンド実行は今後のアップデートで予定されています。

C++ ビルドツールの不足 — 一部のネイティブ依存関係（Node.js のネイティブアドオン等）のビルドに Visual Studio Build Tools（C++ workload）や Microsoft Visual C++ Redistributable (x64) が必要な場合があります。winget install --id Microsoft.VisualStudio.2022.BuildTools -e でインストール後、VS Code を完全に再起動してください。

クラウドタスクのモデル変更不可 — クラウドスレッドのデフォルトモデルは現在変更できません。ローカルスレッドではモデルの切り替えが可能です。

GPT-5.3-Codex-Spark の制限 — Spark モデルは Pro プラン限定の研究プレビューであり、API からの利用やクレジット購入による利用はできません。専用の低レイテンシハードウェア上で動作するため、使用量は需要に応じて調整されます。

Mac版との差異

コア機能（マルチスレッド、ワークツリー、Skills、Automations、Git 統合、MCP、IDE 同期）は Mac 版と同等です。主な差異はサンドボックスの実装方式にあります。Mac版は macOS の Seatbelt ポリシー（sandbox-exec）を使用するのに対し、Windows 版は Restricted Token + ファイルシステムACL + Windows Firewall による独自のサンドボックスを使用します。

Mac 版は Apple Silicon (M1/M2/M3/M4) に最適化されていますが、Windows 版は x64 プロセッサ上で動作します。内蔵ターミナルのデフォルトシェルも Mac は zsh/bash であるのに対し、Windows は PowerShell です。また、Windows 版では WSL を介した Linux エージェントモードの選択肢がある一方、Mac 版にはこの切り替えはありません（Mac は常にネイティブ Unix 環境）。

Section 13 — Tips & Best Practices

Tips・ベストプラクティス

AGENTS.md で指示を効率化する

プロジェクトのルートに AGENTS.md ファイルを配置すると、Codex はこのファイルをプロジェクト固有のガイダンスとして自動的に読み込みます。コーディング規約、ディレクトリ構造の説明、使用技術のバージョン情報、テスト方針などを記載しておくと、毎回のプロンプトで繰り返し指示する手間が省けます。サブディレクトリごとに AGENTS.md をネストすることも可能で、大規模プロジェクトではコンテキスト量を制御するのに役立ちます。

使用量を最適化するテクニック

プロンプトを簡潔かつ正確に — 必要十分な情報を与えつつ、不要なコンテキストは削除しましょう。AGENTS.md が巨大になりすぎると毎回大量のコンテキストが消費されるため、サブディレクトリ分割で管理するのが効果的です。

MCP サーバーを必要時のみ有効に — 接続する MCP サーバーが増えるほどコンテキストも増加します。使わないサーバーは無効にしておくと使用量を節約できます。

ルーチンタスクには Mini モデルを — GPT-5.1-Codex-Mini に切り替えると、使用量制限が約4倍に延長されます。テスト生成やボイラープレートコードの作成など、比較的単純なタスクに最適です。

ワークツリーを活用する — 実験的な変更はワークツリーモードで行い、メインブランチを汚さないようにします。失敗してもワークツリーを削除するだけで済みます。

バージョン管理のベストプラクティス

フィーチャーブランチ上で作業し、Codex に委任する前に git status がクリーンであることを確認します。こまめにコミットすることで、Codex の提案を小さな単位で検証・ロールバックできます。Codex の変更は他の PR と同様にレビューし、コミットメッセージに意思決定の記録を残すことが監査の観点からも重要です。

Skills の活用

OpenAI が提供するキュレーション済みスキルのほか、チーム独自のスキルを作成してリポジトリにチェックインすると、チーム全体で共有できます。1つのスキルには1つの責務を持たせ、スクリプトは決定論的な動作やツール連携が必要な場合にのみ使用し、そうでなければ指示（instructions）で十分です。アプリ内の Skills セクションからスキルの発見・管理・テストができます。

Automations の実践例

OpenAI 社内では、日次のイシュートリアージ、CI 障害の検出・要約、リリースブリーフの自動生成、バグの定期チェック、スキルの自動改善といったタスクに Automations を活用しています。スキルと組み合わせることで、たとえば「毎朝9時に Sentry のエラーをチェックし、P1レベルの問題があれば修正PRを自動作成する」といったワークフローが構築できます。結果はインボックスに届き、報告すべきものがなければ自動アーカイブされます。

パーソナリティの使い分け

Codex には2つのパーソナリティスタイルがあります。terse（簡潔）スタイルは実行重視で無駄なく結果を返し、conversational（会話的）スタイルはより対話的で説明が丁寧です。スレッド内で /personality コマンドを使って切り替えられます。デバッグやリファクタリングなど結果重視のタスクには terse、コードの学習や設計相談には conversational が向いています。

プロ向けTips
config.toml でプロファイルを定義しておくと、codex --profile <name> で瞬時にサンドボックス設定を切り替えられます。たとえば安全重視の readonly_quiet プロファイルと生産性重視の full_auto プロファイルを使い分けると効率的です。
config.toml — プロファイル定義例 [profiles.full_auto] approval_policy = "on-request" sandbox_mode = "workspace-write" [profiles.readonly_quiet] approval_policy = "never" sandbox_mode = "read-only" [profiles.trusted_dev] approval_policy = "untrusted" sandbox_mode = "workspace-write" [profiles.trusted_dev.sandbox_workspace_write] network_access = true
